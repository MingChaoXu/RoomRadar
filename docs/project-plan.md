# 酒店竞品价格爬虫项目方案

更新时间：2026-03-22

## 1. 项目背景

目标是为单体酒店或酒店集团提供一套竞品价格监测工具，围绕目标酒店自动发现周边竞品酒店，并持续采集携程、美团等 OTA 平台的价格，最终输出可用于运营、收益管理和调价决策的竞品价格情报。

本方案的重点不是单次网页抓取，而是建设一个可持续运行的“竞品发现 + 多平台价格采集 + 房型标准化 + 报表分析”系统。

## 2. 项目目标

### 2.1 业务目标

1. 输入一家酒店，自动识别周边竞品酒店列表。
2. 每日定时采集未来多个日期窗的竞品价格。
3. 形成携程、美团两个平台的可比价格数据。
4. 输出价差、均价、最低价、趋势和异常波动告警。

### 2.2 MVP 目标

第一版只做以下闭环：

1. 用户输入酒店名称、地址或经纬度。
2. 通过高德地图 MCP 识别周边 20 到 50 家候选竞品酒店。
3. 采集携程、美团指定入住日期下的最低可售价格。
4. 输出 Excel 和 Web 报表。
5. 支持每日定时更新。

## 3. 核心结论

### 3.1 推荐路线

推荐采用“三层适配”方案：

1. 高德 MCP 负责目标酒店定位、周边竞品发现、POI 标准化。
2. OTA 价格采集层采用 `官方 API 优先 + 浏览器自动化兜底 + 第三方聚合接口备用`。
3. 中台数据层负责房型映射、价格口径统一、历史快照存储和分析输出。

### 3.2 为什么不建议第一版完全依赖官方 API

调研显示，Trip/携程存在合作方酒店可用性查询接口，但需要合作方参数和签名，例如 `aid`、`sid`、`token`、`icode`、`uuid`，明显是伙伴接入模式，而不是个人开发者即可调用的公开查询接口。Trip partner API 文档给出了接口地址、鉴权参数和请求示例。来源：Trip API 文档 <https://apidoc.trip.com/apidoc>

美团酒店直连平台同样有标准对外接口，但官方流程页明确写明：当前“暂不接入 PMS 类供应商，仅与连锁酒店或酒店代理直连”。这意味着独立 SaaS 或酒店运营工具在没有商务合作资质时，不能把美团官方接口当成 MVP 唯一来源。来源：美团酒店直连平台流程页 <https://openplatform-hotel.meituan.com/portal/process>

因此，第一版应假设“暂时拿不到官方授权”，优先建设可插拔采集层。

## 4. 调研结论

### 4.1 高德地图与 MCP 的可行性

高德地图 MCP Server 已有开源实现，并公开支持 `maps_text_search`、`maps_around_search`、`maps_search_detail` 等工具，足以完成以下任务：

1. 通过酒店名称定位目标酒店。
2. 通过周边搜索获取酒店周边竞品候选集。
3. 通过 POI 详情补充地址、电话、经纬度、类型等字段。

参考项目：

1. sugarforever/amap-mcp-server  
   <https://github.com/sugarforever/amap-mcp-server>
2. JavaAIDev/amap-mcp-server  
   <https://github.com/JavaAIDev/amap-mcp-server>

### 4.2 携程价格采集可行性

携程官方存在合作方酒店可用性 API，但适用对象更像联盟、分销、合作伙伴，而不是普通采集应用。对于没有合作资质的项目，现实路径通常是：

1. 浏览器自动化采集 Web/H5 页面。
2. 抓取页面 XHR/GraphQL/JSON 数据接口。
3. 若反爬成本过高，则接入第三方聚合数据服务。

官方合作接口参考：

1. Trip partner API 文档  
   <https://apidoc.trip.com/apidoc>

### 4.3 美团价格采集可行性

美团官方开放方向更偏“酒店供应接入”和“库存房态直连”，不适合作为独立竞品监测工具直接调用。没有合作资格时，也应视为浏览器自动化采集或第三方接口的场景。

官方资料参考：

1. 美团酒店直连平台  
   <https://openplatform-hotel.meituan.com/>
2. 接入流程  
   <https://openplatform-hotel.meituan.com/portal/process>

### 4.4 开源项目现状

没有发现一个成熟、持续维护、并且直接针对“高德定位 + 携程/美团竞品价格监控”的完整开源方案。现成项目大多分为以下几类：

1. 地图检索类 MCP。
2. 通用爬虫框架。
3. 酒店搜索 API/MCP 示例。
4. Google Hotels 或 Booking 等境外平台价格采集示例。

能直接借鉴的开源项目包括：

1. `apify/crawlee`  
   Node.js/TypeScript 爬虫框架，支持 Playwright、代理轮换、会话管理、重试、队列和存储。  
   <https://github.com/apify/crawlee>
2. `apify/crawlee-python`  
   Python 版本，适合和 FastAPI、数据分析栈整合。  
   <https://github.com/apify/crawlee-python>
3. `MFori/google-hotels-scraper`  
   演示了“指定入住离店日期 + 多提供商价格抓取”的产品形态，可借鉴其输出结构。  
   <https://github.com/MFori/google-hotels-scraper>
4. `esakrissa/hotels_mcp_server`  
   用 Booking API 做酒店搜索和价格查询的 MCP 示例，可借鉴 MCP 工具封装方式。  
   <https://github.com/esakrissa/hotels_mcp_server>
5. `justoneapi/justoneapi-python`  
   提供携程、美团、高德等平台的商业化聚合接口 SDK 线索，但需要自行评估数据质量、成本和合规性。  
   <https://github.com/justoneapi/justoneapi-python>

## 5. 产品方案

### 5.1 角色定义

1. 运营经理：查看竞品价格、价差、趋势、异常。
2. 收益经理：按日期、房型、平台做调价分析。
3. 酒店老板/总经理：看核心指标和日报。
4. 技术管理员：配置采集任务、代理池、账号、告警和重试策略。

### 5.2 典型使用场景

1. 指定一家酒店，查看 3 公里内竞品未来 30 天最低价。
2. 查看节假日前后竞品价格变化。
3. 发现“某竞品在携程突然降价 20%”并推送告警。
4. 对比自家酒店在不同平台上的竞争力区间。

### 5.3 产品功能清单

#### A. 酒店基础管理

1. 录入目标酒店。
2. 酒店信息标准化。
3. 城市、商圈、品牌、星级标签管理。

#### B. 竞品发现

1. 高德定位目标酒店。
2. 周边半径搜索酒店候选。
3. 按品牌、距离、档次过滤。
4. 人工确认竞品池。

#### C. OTA 采集

1. 携程价格采集。
2. 美团价格采集。
3. 多日期窗采集。
4. 多入住人数和早餐条件采集。
5. 采集失败自动重试。

#### D. 标准化引擎

1. 酒店匹配。
2. 房型映射。
3. 价格口径标准化。
4. 促销价、会员价、返现价字段拆分。

#### E. 分析报表

1. 竞品最低价排行。
2. 竞品均价趋势。
3. 自家与竞品价差。
4. 平台价差。
5. 日期维度异常波动预警。

#### F. 运维能力

1. 任务调度。
2. 失败任务重试。
3. 代理池与风控控制。
4. 日志、审计与采集截图存档。

## 6. 系统架构

### 6.1 总体架构

```text
┌──────────────────────────────────────┐
│              管理后台 / BI           │
└──────────────────────────────────────┘
                    │
┌──────────────────────────────────────┐
│            应用服务层 FastAPI         │
│  酒店管理 | 竞品管理 | 查询接口 | 告警  │
└──────────────────────────────────────┘
                    │
┌──────────────────────────────────────┐
│            任务与调度层               │
│ APScheduler / Celery / Redis Queue   │
└──────────────────────────────────────┘
                    │
┌──────────────────────────────────────┐
│             采集适配层                │
│ Amap Adapter | Ctrip Adapter | MT Adapter │
│ Official API | Browser | Fallback API │
└──────────────────────────────────────┘
                    │
┌──────────────────────────────────────┐
│           标准化与规则引擎             │
│ 酒店匹配 | 房型匹配 | 价格清洗 | 告警规则 │
└──────────────────────────────────────┘
                    │
┌──────────────────────────────────────┐
│             数据存储层                │
│ PostgreSQL | Redis | Object Storage  │
└──────────────────────────────────────┘
```

### 6.2 模块说明

#### 6.2.1 Amap Adapter

职责：

1. 将用户输入的酒店名称/地址标准化为经纬度。
2. 获取周边酒店 POI 列表。
3. 补充 POI 详情和结构化地址。

#### 6.2.2 OTA Adapter

每个平台独立封装，遵循统一接口：

```python
class PriceCollectorAdapter(Protocol):
    async def search_hotel(self, query: HotelSearchQuery) -> list[HotelCandidate]: ...
    async def fetch_price(self, query: RateQuery) -> list[RoomRateSnapshot]: ...
```

每个平台至少实现 3 套能力：

1. `official_adapter`
2. `browser_adapter`
3. `fallback_adapter`

#### 6.2.3 标准化引擎

职责：

1. 将高德 POI、携程酒店、美团酒店映射到统一酒店主数据。
2. 将 OTA 原始房型名称归一成标准房型。
3. 拆分价格维度，形成可比较字段。

#### 6.2.4 报表与告警

职责：

1. 生成日级价格快照报表。
2. 生成周报、月报。
3. 触发价格异常通知。

## 7. 技术选型建议

### 7.1 推荐技术栈

后端推荐 Python：

1. `FastAPI`：接口层和后台管理 API。
2. `Playwright`：动态页面采集。
3. `PostgreSQL`：主数据与价格快照。
4. `Redis`：任务队列、缓存、去重、限频。
5. `APScheduler` 或 `Celery`：定时任务。
6. `Pydantic`：请求和采集结果模型。
7. `Pandas`：规则清洗和导出。

原因：

1. 与数据清洗、报表输出结合更自然。
2. 便于后续做简单预测、价格分析和模型扩展。
3. 团队若以数据工程为主，维护成本低。

### 7.2 可选技术栈

如果团队前端/Node 更强，可选：

1. `Crawlee + Playwright + NestJS`

优点：

1. Crawlee 的浏览器指纹、会话、代理轮换、失败重试比较成熟。
2. 更适合把采集系统拆成独立 worker。

## 8. 关键流程设计

### 8.1 竞品发现流程

1. 用户输入目标酒店名称或地址。
2. 通过 Amap MCP 进行关键词搜索定位目标酒店。
3. 调用周边搜索获取指定半径的酒店列表。
4. 按品牌、星级、距离、点评量做候选排序。
5. 人工确认最终竞品池。

### 8.2 价格采集流程

1. 任务调度器生成采集计划。
2. OTA Adapter 按日期、入住人数、渠道参数进行查询。
3. 解析页面或接口返回的房型与价格。
4. 存储原始结果。
5. 触发标准化引擎做酒店匹配和房型映射。
6. 写入价格快照表。
7. 更新报表缓存和告警状态。

### 8.3 房型标准化流程

1. 对 OTA 房型名称做中文分词和规则拆解。
2. 识别床型、面积、窗型、早餐、取消规则。
3. 映射到标准房型模板。
4. 置信度不足时进入人工校验。

## 10. 风险与应对

### 10.1 反爬与稳定性风险

风险：

1. OTA 页面存在动态渲染、风控验证、频率限制。
2. 价格和库存依赖日期、设备、账号、地域和优惠策略。

应对：

1. 采集层与解析层解耦。
2. 保留截图、HTML、接口响应原文，方便回放。
3. 加代理池、限速、随机化和会话管理。
4. 采集计划按城市和平台分片执行。

### 10.2 数据可比性风险

风险：

1. 房型名称不统一。
2. 价格可能存在“会员价”“券后价”“含税/未税价”差异。

应对：

1. 建立标准房型字典。
2. 价格维度拆成多个字段而不是只存一个“price”。
3. 前端展示必须标明价格口径。

### 10.3 合规与商务风险

风险：

1. 官方开放接口通常要求合作资质。
2. 直接网页采集可能受到平台限制或风控。

应对：

1. 方案上支持官方 API 接入。
2. 采集能力做成 adapter，便于后续替换为商务合作接口。
3. 项目上线前让法务或合规同事审查目标平台条款和采集策略。

注：这里的合规判断是工程建议，不构成法律意见。

## 11. 项目实施建议

### 11.1 第一阶段

先做“可跑通、可验证价值”的 MVP：

1. 仅支持单城市。
2. 仅支持携程、美团两个平台。
3. 仅支持标准入住场景：1 间房、2 成人。
4. 仅输出最低价和基础房型信息。

### 11.2 第二阶段

在 MVP 稳定后扩展：

1. 多城市批量采集。
2. 多酒店集团权限隔离。
3. 会员价、促销价识别。
4. 价格预测和建议调价。
